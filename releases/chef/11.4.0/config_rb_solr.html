solr.rb¶

Note

This topic applies to Chef 0.10.x.

A solr.rb file is used to specify the configuration details for chef-solr, chef-solr-indexer, and chef-solr-rebuild. This file is the default configuration file and is loaded every time these executable are run. The chef-solr and chef-solr-indexer executables can be run as a daemon. (The chef-solr-rebuild executable cannot be run as an daemon.) The configuration file is located at: /etc/chef/solr.rb.

Settings¶

This configuration file has the following settings:

Setting Description
amqp_consumer_id

The identifier for the consumer that will handle messages. For example:

amqp_consumer_id "default"
amqp_host

The IP address for the host to which a connection is made. For example:

amqp_host 0.0.0.0
amqp_pass

The password for the amqp_user. For example:

amqp_pass 12345678
amqp_port

The port number for the host to which a connection is made. For example:

amqp_port 1234
amqp_user

A user name with permission to access the host. For example:

amqp_user adamjacobs
amqp_vhost

The virtual host name. For example:

amqp_vhost "/chef"
authorized_openid_identifiers

The identifier for the OpenID associated with the Chef Server web UI. For example:

username.myopenid.com

or:

http://username.myopenid.com
authorized_openid_providers

The provider for the OpenID associated with the Chef Server web UI. For example:

myopenid.com
cache_options

The option(s) to be used when caching. The options available depend on the type of cache being used. For example:

cache_options({
  :path => "/var/chef/cache/checksums",
  :skip_expires => true
})
cache_type

The type of cache to use. This can be any type of cache that is supported by the Moneta library (https://github.com/wycats/moneta), including basic file stores, file systems with xattrs, DataMapper, Amazon S3, Berkeley DB, MongoDB, Redis, SBDM, Tokyo Cabinet, CouchDB, and so on. Default value: BasicFile. For example:

cache_type "BasicFile"
client_key

The location of the file which contains the client key. Default value: /etc/chef/client.pem. For example:

chef_server_url "http://localhost:4000"
chef_server_url

The URL for the Chef Server. For example:

chef_server_url "http://localhost:4000"
client_registration_retries

The number of times a chef-client should attempt to register with a Chef Server. Default value: 5. For example:

client_registration_retries 5
client_url

The URL that is used by a chef-client to connect to a Chef Server. Default value: http://localhost:4042. For example:

client_url "http://localhost:4042"
cookbook_path

The sub-directory for cookbooks on the chef-client. This value can be a string or an array of file system locations, processed in the specified order. The last cookbook is considered to override local modifications. For example:

cookbook_path [
  "/var/chef/cookbooks",
  "/var/chef/site-cookbooks"
]
cookbook_tarball_path

The location in which a Chef Server stores cookbooks that have been saved in the .tar file format, and from which cookbooks are distributed to a chef-client. Default value: /var/chef/cookbook-tarballs. For example:

cookbook_tarball_path "/var/chef/cookbook-tarballs"
couchdb_database

The name of the CouchDB database. For example:

couchdb_database "chef"
couchdb_url

The URL for the CouchDB database. For example:

couchdb_url "http://localhost:5984"
couchdb_version

Optional. The version of the CouchDB database. For example:

couchdb_version nil
data_bag_path

The location from which a data bag is loaded to Chef. Default value: /var/chef/databags. For example:

data_bag_path "/var/chef/databags"
environment

The name of the environment. This is typically set from the command line (using the -E option) or in the client.rb configuration file. For example:

environment "production"
file_cache_path

The location in which cookbooks (and other transient data) files are stored when they are synchronized with Chef. (This value can also be used in recipes to download files with the remote_file resource.) For example:

file_cache_path "/var/chef/cache"
file_backup_path

The location in which backup files are stored. If this value is empty, backup files will be stored in the directory of the target file. Default value: /var/chef/backup. For example:

file_backup_path "/var/chef/backup"
group

The group that owns a process. This is required when starting any executable as a daemon. Default value: nil. For example:

group nil
http_proxy

The proxy server for HTTP connections. Default value: nil. For example:

http_proxy "http://proxy.vmware.com:3128"
http_proxy_pass

The password for the proxy server. Default value: nil. For example:

http_proxy_pass "1234567890"
http_proxy_user

The user name for the proxy server. Default value: nil. For example:

http_proxy_user "my_username"
http_retry_count

The number of retry attempts. Default value: 5. For example:

http_retry_count 5
http_retry_delay

The delay (in seconds) between retry attempts. Default value: 5. For example:

http_retry_delay 5
https_proxy

The proxy server for HTTPS connections. Default value: nil. For example:

https_proxy "http://proxy.vmware.com:3128"
interval

The interval at which the chef-client is run. This value is configured for the chef-client application run time, rather than in Chef::Config. Default value: 1800. For example:

interval 1800
json_attribs

The path to a file that contains JSON data. Use this option to override attributes that are set from other locations, such as from within a cookbook or by a role. For example:

json_attribs nil
log_level

The level of logging that will be stored in a log file: debug, info, warn, error, or fatal. For example:

log_level :info
log_location

The location in which log file output files will be saved. If this location is set to something other than STDOUT, standard output logging will still be performed (otherwise there would be no output other than to a file). Default value: STDOUT. For example:

log_location STDOUT
node_name

The name of the node. This is used to determine which configuration should be applied and to set the client_name (which is the name used when authenticating to a Chef Server). The default value is set automatically to be the fully qualified domain name (FQDN) of the chef-client, as detected by Ohai. For example:

node_name "mynode.example.com"
node_path

The location in which to look for node-specific recipes. Default value: /var/chef/node. For example:

node_path "/var/chef/node"
no_proxy

A comma-separated list of URLs that do not need a proxy. Default value: nil. For example:

no_proxy "*.vmware.com,10.*"
openid_cstore_couchdb

Indicates whether OpenID data is stored in CouchDB (instead of the file system). Default value: false. For example:

openid_cstore_couchdb false
openid_cstore_path

The location in which OpenID data is stored in the file system. Default value: /var/chef/openid/cstore. For example:

openid_cstore_path "/var/chef/openid/cstore"
pid_file

The location in which a process identification number (pid) is saved. An executable, when started as a daemon, will write the pid to the specified file. Default value: /tmp/name-of-executable.pid. For example:

pid_file "/tmp/executable.pid"
recipe_url

The URL location from which a remote cookbook tar.gz will be downloaded. For example:

recipe_url "http://path/to/remote/cookbook"
rest_timeout

The time (in seconds) after which an HTTP REST request will time out. Default value: 300. For example:

rest_timeout 300
role_path

The location in which role files are located. Default value: /var/chef/roles. For example:

role_path "/var/chef/roles"
signing_ca_cert

The path to where the certificate is located. This setting is generated automatically by Chef and is passed to OpenSSL. Default value: /var/chef/ca/cert.pem. For example:

signing_ca_cert "/var/chef/ca/cert.pem"
signing_ca_key

The path to where the private key is located. This setting is generated automatically by Chef and is passed to OpenSSL. Default value: /var/chef/ca/key.pem. For example:

signing_ca_key "/var/chef/ca/key.pem"
signing_ca_user

The user associated with the certificate authority (CA) policy. This setting is passed by Chef to OpenSSL. For example:

signing_ca_user nil
signing_ca_group

The group associated with the certificate authority (CA) policy. This setting is passed by Chef to OpenSSL. For example:

signing_ca_group nil
signing_ca_country

The country code associated with the certificate authority (CA) policy. This setting is passed by Chef to OpenSSL. For example:

signing_ca_country "US"
signing_ca_state

The state (or province) associated with the certificate authority (CA) policy. This setting is passed by Chef to OpenSSL. For example:

signing_ca_state "Washington"
signing_ca_location

The location associated with the certificate authority (CA) policy. This setting is passed by Chef to OpenSSL. For example:

signing_ca_location "Seattle"
signing_ca_org

The organization associated with the certificate authority (CA) policy. This setting is passed by Chef to OpenSSL. For example:

signing_ca_org "Chef User"
signing_ca_domain

The domain associated with the certificate authority (CA) policy. This setting is passed by Chef to OpenSSL. For example:

signing_ca_domain "opensource.opscode.com"
signing_ca_email

The email address associated with the certificate authority (CA) policy. This setting is passed by Chef to OpenSSL. Default value: opensource.opscode.com. For example:

signing_ca_email "opensource-cert@opscode.com"
solo

Indicates whether Chef is being run in “solo” mode. This setting determines if the chef-client should attempt to communicate with a Chef Server. Default value: false. For example:

solo false
solr_data_path``.

The location in which the Solr indexes are located. Default value: /var/chef/solr/data. For example:

solr_data_path "/var/chef/solr/data"
solr_heap_size

The amount of memory allocated to the Java Virtual Machine (JVM) that is being used to run Solr. Default value: 256M. For example:

solr_heap_size 256M
solr_home_path

The path to the Solr home directory. Default value: /var/chef/solr. For example:

solr_home_path "/var/chef/solr"
solr_java_opts

The additional options (if any) that are passed to the Solr JVM. Default value: nil. For example:

solr_java_opts nil
solr_jetty_path

The path to the settings that control the Jetty environment. Default value: /var/chef/solr-jetty. For example:

solr_jetty_path "/var/chef/solr-jetty"
solr_url

The URL for the server from which the Solr search engine is run. Default value: http://localhost:8983. For example:

solr_url "http://localhost:8983"
splay

A number (in seconds) to add to the interval that is used to determine the frequency of chef-client runs. This number can help prevent server load when there are many clients running at the same time. Default value: nil. For example:

splay
ssl_client_cert

The OpenSSL X509 certificate. This setting is generated automatically by Chef and most users will not need to modify it. For example:

ssl_client_cert ""
ssl_client_key

The OpenSSL X509 key. This setting is generated automatically by Chef and most users will not need to modify it. For example:

ssl_client_key ""
ssl_ca_path

The path to where the OpenSSL key is located. This setting is generated automatically by Chef and most users will not need to modify it. For example:

ssl_ca_path nil "/etc/ssl/certs"
ssl_ca_file

The file in which the OpenSSL key is saved. This setting is generated automatically by Chef and most users will not need to modify it. For example:

ssl_ca_file nil
ssl_verify_mode

The verify mode for HTTPS requests. Set to :verify_none to not send certificate requests. Set to :verify_peer to send certificate requests to a chef-client. For users of Hosted Chef, :verify_peer is recommended as the verification mode; depending on how OpenSSL is configured, the ssl_ca_path may need to be specified. For example, on an Ubuntu system:

ssl_verify_mode :verify_peer
umask

The file mode creation mask. Default value: 0022. For example:

umask 0022
user

The user that owns a process. This is required when starting any executable as a daemon. Default value: nil. For example:

user nil
validation_key

The location of the file which contains the key used when a chef-client is registered with a Chef Server. A validation key is signed using the validation_client_name for authentication. Default value: /etc/chef/validation.pem. For example:

validation_key "/etc/chef/validation.pem"
validation_client_name

The name of the server that–along with the validation_key–is used to determine whether a chef-client may register with a Chef Server. The validation_client_name located in the server and client configuration files must match. For example:

validation_client_name "chef-validator"
verbose_logging

Indicates the level of logging. Options: true, nil, and false. When this is set to false, notifications about individual resources being processed will be suppressed (and will be output at the :info logging level). Setting this to false can be useful when a chef-client is run as a daemon. Default value: nil. For example, when verbose_logging is set to true or nil:

[date] INFO: *** Chef 0.10.6.rc.1 ***
[date] INFO: Setting the run_list
             to ["recipe[a-verbose-logging]"] from JSON
[date] INFO: Run List is [recipe[a-verbose-logging]]
[date] INFO: Run List expands to [a-verbose-logging]
[date] INFO: Starting Chef Run for some_node
[date] INFO: Running start handlers
[date] INFO: Start handlers complete.
[date] INFO: Loading cookbooks [test-verbose-logging]
[date] INFO: Processing file[/tmp/a1] action create
             (a-verbose-logging::default line 20)
[date] INFO: Processing file[/tmp/a2] action create
             (a-verbose-logging::default line 21)
[date] INFO: Processing file[/tmp/a3] action create
             (a-verbose-logging::default line 22)
[date] INFO: Processing file[/tmp/a4] action create
             (a-verbose-logging::default line 23)
[date] INFO: Chef Run complete in 1.802127 seconds
[date] INFO: Running report handlers
[date] INFO: Report handlers complete

When verbose_logging is set to false (for the same output):

[date] INFO: *** Chef 0.10.6.rc.1 ***
[date] INFO: Setting the run_list
             to ["recipe[a-verbose-logging]"] from JSON
[date] INFO: Run List is [recipe[a-verbose-logging]]
[date] INFO: Run List expands to [a-verbose-logging]
[date] INFO: Starting Chef Run for some_node
[date] INFO: Running start handlers
[date] INFO: Start handlers complete.
[date] INFO: Loading cookbooks [a-verbose-logging]
[date] INFO: Chef Run complete in 1.565369 seconds
[date] INFO: Running report handlers
[date] INFO: Report handlers complete

Where in the examples above, [date] represents the date and time the long entry was created. For example: [Mon, 21 Nov 2011 09:37:39 -0800].
web_ui_admin_default_password

The default password for the Chef Server web UI. This password should be changed immediately after logging in for the first time. Default value: p@ssw0rd1. For example:

web_ui_admin_default_password "p@ssw0rd1"
web_ui_admin_user_name

The default user name for the Chef Server web UI. Default value: admin. For example:

web_ui_admin_user_name  "admin"
web_ui_client_name

The name of the Chef Server web UI. Default value: chef-webui. For example:

web_ui_client_name "chef-webui"
web_ui_key

The path to where the key used by the Chef Server web UI is located. Default value: /etc/chef/webui.pem. For example:

web_ui_key "/etc/chef/webui.pem"
